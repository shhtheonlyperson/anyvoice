Current Architecture

AnyVoice is a Vercel-hosted Next app with a local or GPU VoxCPM2 worker boundary. The browser owns capture and consent; the API owns validation, run files, worker forwarding, and audio serving. Streaming/progress endpoints are currently in flight in the worktree, so this page treats them as implementation work that still needs verification.

A Recording modes

The current direction is correct: make scripted capture the default because it gives the model a known transcript. Freeform/upload stays available, but it must require the exact words in the clip.

B Quality contract

The backend now treats prompt transcript as required. This should remain the visible product truth: wrong transcript is the dominant avoidable source of bad pronunciation.

C Streaming reality

The stable output model is file-ready playback, while active stream-route work is adding NDJSON progress. The quality bar is still honest stages first, progressive audio second.

UX Weak Points

The product goal is simple: "record my voice, type any text, generate." The weak points are where necessary model constraints leak as confusion, waiting, or silent failure.

1 Recording confidence arrives too late

The UI can show level and elapsed time, but the strongest reference-quality signals are returned after worker processing. Users need earlier feedback on duration, silence, clipping, and loudness.

2 Transcript friction is justified but fragile

Freeform/upload requiring an exact transcript is the right contract, but it is easy to perceive as busywork unless the UI explains that this is pronunciation alignment, not metadata.

3 Worker-missing can feel like a failed recording

On Vercel or an unconfigured machine, the clone route may return needs_worker. That is true, but after a long capture flow it reads as "the app failed" unless readiness is clear up front.

4 Long synthesis has no stage model

The non-stream path is effectively "submit and wait." The in-flight stream path must prove concrete stages: upload accepted, preprocessing, quality analysis, synthesis, finalizing.

5 Quality knobs should stay mostly backstage

Speed, balanced, and quality presets exist, but adding them as front-page controls would increase cognitive load. The better first move is smarter defaults plus transparent effective metadata.

6 Streaming is a product promise, not a spinner style

Real streaming means the backend can emit usable audio chunks or segments before the full run is complete. Until then, the honest UX is job progress and fast playback once the file is ready.

Recommended Plan

Lead with reliability and quality feedback. Only add deeper streaming once the job lifecycle is explicit enough to verify.

Rationale And Tradeoffs

These are the decisions most likely to be reopened during implementation. The recommendation is opinionated because the model has sharp constraints.

Scripted transcript vs automatic transcript

Why scripted wins now

It makes the prompt transcript exact by construction and avoids turning ASR errors into TTS pronunciation errors.

Cost

It asks the user to read a line. This is acceptable because it directly buys synthesis quality.

Warn vs block on poor reference audio

Warn first

Users should be allowed to test imperfect clips, especially during debugging and local model work.

Block only hard invalids

Empty recording, missing consent, missing transcript, and oversized uploads are contract failures.

Progress states vs true streaming

Progress states first

They are verifiable and map to real work: upload, preprocess, analyze, synthesize, finalize.

Streaming later

Segment-ready audio is only worth it if quality and continuity survive chunking.

Preset controls vs default quality

Keep simple UI

Balanced should remain default. Users came to clone a voice, not tune diffusion params.

Keep metadata honest

Store effective timesteps, cfg, denoise, quality preset, and reference grade for debugging.

Decision rule

If a UI element helps the user produce a better reference, understand synthesis progress, or recover from a failed run, keep it. If it only exposes runtime trivia, move it to metadata or docs.

Verification Checklist

Use this as the acceptance bar for the implementation agent's changes. It separates recording, synthesis quality, streaming UX, and product safety.

R Recording

  • Mic permission denied, no microphone, and unsupported recorder states show actionable copy.
  • Scripted recording creates a usable File and pairs it with the known script transcript.
  • Freeform/upload requires transcript before submit.
  • Captured source audio is playable before generation.
  • Duration, loudness, and likely clipping/silence guidance appear before submit.
  • 60 second cap is visible while recording and stops cleanly.

Q Synthesis Quality

  • Backend receives promptTranscript on every valid request.
  • VoxCPM2 call stays in ultimate mode with verified prompt text.
  • Generated output does not prepend the prompt audio.
  • Metadata includes reference quality and effective params.
  • Known scripted Mandarin fixture remains intelligible and speaker-like.
  • Bad reference warnings map to user-readable recommendations.

S Streaming And Latency

  • Submit immediately confirms request acceptance or validation failure.
  • Long generation shows real stage transitions, not a decorative spinner alone.
  • NDJSON progress, if used, handles partial lines, worker proxying, final ready payload, and error payloads.
  • Worker-missing state appears before the user assumes the recording failed.
  • Audio playback starts from the returned URL once the file is ready.
  • Any segment-ready experiment is compared against whole-file quality before shipping.
  • Retry path does not require re-recording when the reference is still valid.

P Product And Safety

  • Consent remains a hard API gate and visible near the generate action.
  • zh-Hant remains default and English remains available.
  • Light, dark, and system theme controls still work.
  • Keyboard navigation reaches recording, upload, transcript, consent, and submit controls.
  • No runtime metadata overwhelms the primary "record, type, generate" workflow.
  • Vercel preview, local worker, and protected Mac Studio worker paths are tested separately.
